DistributedDataSource

Shard-aware datasource that keeps the regular IDataSource surface while routing reads/writes across multiple IProxyCluster shards with resilient execution, observability, audit, security, and backpressure controls.

Overview

DistributedDataSource composes existing shard clusters (IProxyCluster) into one logical datasource. You keep calling normal datasource methods, but distribution strategy is driven by a DistributionPlan and runtime options.

ConcernHandled ByWhat it gives you
Shard routingDistributionPlan + resolver/routerSharded/replicated/broadcast placement decisions
Per-shard HAIProxyClusterFailover/circuit/load-balancing inside each shard
Cross-shard executionDistributed executorsScatter, replicated reads, fan-out writes
ObservabilityMetrics + eventsHot shard signals, degraded mode, operation telemetry
Capacity controlsPhase 14 performance blockConcurrency caps, rate limits, adaptive deadlines, backpressure

Construction Pattern

using TheTechIdea.Beep.Distributed;
using TheTechIdea.Beep.Distributed.Performance;
using TheTechIdea.Beep.Proxy;

IDMEEditor editor = /* existing editor */;

IReadOnlyDictionary<string, IProxyCluster> shards = new Dictionary<string, IProxyCluster>
{
    ["shard-a"] = shardACluster,
    ["shard-b"] = shardBCluster,
    ["shard-c"] = shardCCluster
};

DistributionPlan plan = /* load from plan store or build */;

var options = new DistributedDataSourceOptions
{
    DefaultPerShardTimeoutMs = 15_000,
    DefaultReadDeadlineMs    = 45_000,
    MaxScatterParallelism    = 8,
    MaxFanOutParallelism     = 6
};

var distributed = new DistributedDataSource(
    dmeEditor: editor,
    shards:    shards,
    plan:      plan,
    options:   options);

distributed.Openconnection();
Use explicit shard IDs and keep them stable over time. Plan routing and operational logs depend on those IDs.

Clear Examples

Example 1 — Single-key read routed to one shard

// Standard IDataSource call style remains valid.
var customer = distributed.GetEntity("Customers", new List<AppFilter>
{
    new AppFilter { FieldName = "CustomerId", Operator = "=", FilterValue = customerId }
});

Example 2 — Replicated read with fallback

// For replicated placement, resolver picks an ordered replica set.
// Executor attempts replicas according to policy and shard health.
var product = distributed.GetEntity("Products", new List<AppFilter>
{
    new AppFilter { FieldName = "Sku", Operator = "=", FilterValue = sku }
});

Example 3 — Sharded write

var order = new
{
    OrderId = Guid.NewGuid(),
    CustomerId = customerId, // routing key in most plans
    Amount = 149.99m,
    Status = "New"
};

IErrorsInfo writeResult = distributed.InsertEntity("Orders", order);
if (writeResult.Flag != Errors.Ok)
{
    throw new InvalidOperationException(writeResult.Message);
}

Example 4 — Backpressure-aware call path

using TheTechIdea.Beep.Distributed.Performance;

try
{
    var rows = distributed.GetEntity("Orders", null);
}
catch (BackpressureException bp)
{
    // Retry after the provided hint.
    logger.LogWarning(
        $"Distributed gate '{bp.GateName}' rejected call. Retry after {bp.RetryAfter.TotalMilliseconds:F0} ms.");
}

Example 5 — Subscribe to operational events

distributed.OnShardDown += (_, e) =>
    logger.LogWarning($"Shard down: {e.ShardId}");

distributed.OnShardRestored += (_, e) =>
    logger.LogInformation($"Shard restored: {e.ShardId}");

distributed.OnDegradedMode += (_, e) =>
    logger.LogWarning($"Degraded mode. Healthy ratio={e.HealthyRatio:F2}");

distributed.OnBackpressure += (_, e) =>
    logger.LogWarning(
        $"Backpressure gate={e.GateName} shard={e.ShardId ?? "-"} retry={e.RetryAfter.TotalMilliseconds:F0}ms");

Operational Events

Events to wire for observability and alerting:

  • OnShardSelected — routing telemetry.
  • OnPlacementViolation — plan violations or invalid placement.
  • OnShardDown / OnShardRestored — shard health transitions.
  • OnDegradedMode — healthy ratio below threshold.
  • OnPartialReplicationFailure / OnPartialBroadcast — quorum succeeded with gaps.
  • OnBackpressure — Phase 14 capacity rejection events.

Phase 14 Capacity Controls

var options = new DistributedDataSourceOptions
{
    Performance = new PerformanceOptions
    {
        EnableCapacityGates          = true,
        MaxConcurrentDistributedCalls = 64,
        DistributedPermitWait        = TimeSpan.FromMilliseconds(500),
        MaxConcurrentCallsPerShard   = 16,
        ShardPermitWait              = TimeSpan.FromMilliseconds(300),
        ShardRateLimitPerSecond      = 120,
        ShardRateLimitBurst          = 240,
        AdaptiveDeadlineFactor       = 1.3,
        MinAdaptiveDeadlineMs        = 250,
        MaxAdaptiveDeadlineMs        = 8_000,
        EnableHotShardReadShedding   = true
    }
};

This layer runs before cluster-tier execution, so overload is rejected early with a structured BackpressureException instead of hidden latency growth.

Common Pitfalls

  • Using empty or unstable shard IDs. Keep IDs deterministic and environment-consistent.
  • Running with a plan that doesn't match actual shard map. Validate before deployment.
  • Overly high fan-out/scatter parallelism. Start conservative, measure, then scale up.
  • Ignoring backpressure events. Wire OnBackpressure to metrics/alerts.
  • Treating distributed writes as always all-or-nothing without checking partial failure events.
