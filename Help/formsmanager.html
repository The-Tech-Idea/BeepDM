Overview

FormsManager is the BeepDM form-orchestration runtime in the TheTechIdea.Beep.Editor.UOWManager namespace. It coordinates block registration, current-block state, navigation, query-mode and CRUD-mode transitions, master/detail synchronization, triggers, LOV application, savepoints, locks, audit capture, security enforcement, paging, cache control, and multi-form communication.

The current public runtime surface is documented here under the FormsManager name.

What this page covers
  • Current public runtime surface and partial-class layout
  • Caller-facing helper managers and orchestration boundaries
  • Mode transitions, CRUD flows, and built-in runtime operations
  • Examples for explicit keys, sequences, identity keys, defaults, audit, security, and paging

Helper Manager Surface

FormsManager exposes most subsystem capabilities through helper properties so callers can use low-level behavior without re-implementing orchestration. In normal application code, start with FormsManager entry points and reach for helper managers only when you need custom registration or inspection behavior.

Practical rule

Use FormsManager methods for orchestration entry points like commit, rollback, query, navigation, and LOV application. Use helper managers for registration, inspection, custom rule wiring, or host-specific integration.

Form Lifecycle and Mode Transitions

The normal lifecycle is open form, register blocks, enter query mode when needed, execute a query, navigate, edit, and commit or rollback through a single form-level orchestration point.

Lifecycle example

var manager = new FormsManager(dmeEditor)
{
    CurrentFormName = "CUSTOMER_ORDERS"
};

await manager.OpenFormAsync("CUSTOMER_ORDERS");
await manager.EnterQueryModeAsync("CUSTOMERS");

var filters = new List<AppFilter>
{
    new AppFilter { FieldName = "Country", Operator = "=", FilterValue = "USA" }
};

await manager.ExecuteQueryAndEnterCrudModeAsync("CUSTOMERS", filters);
await manager.FirstRecordAsync("CUSTOMERS");
await manager.NextRecordAsync("CUSTOMERS");

var result = await manager.CommitFormAsync();
if (result.Flag != Errors.Ok)
{
    await manager.RollbackFormAsync();
}

Enter query modeEnterQueryModeAsync(string blockName)Switches the target block into query mode
Execute query and return to CRUD modeExecuteQueryAndEnterCrudModeAsync(...)Runs filters, reloads block data, then restores normal editing mode

Block Registration and CRUD Flow

Prefer typed registration so FormsManager can create records with the correct CLR type, use typed accessors, and keep new-record flows compatible with validation, LOVs, audit, and key generation.

Typed block registration and master-detail wiring

manager.RegisterBlock<CustomerDto>(
    blockName: "CUSTOMERS",
    unitOfWork: customerUow,
    entityStructure: customerEntity,
    dataSourceName: "Northwind",
    isMasterBlock: true);

manager.RegisterBlock<OrderDto>(
    blockName: "ORDERS",
    unitOfWork: orderUow,
    entityStructure: orderEntity,
    dataSourceName: "Northwind");

manager.CreateMasterDetailRelation(
    masterBlockName: "CUSTOMERS",
    detailBlockName: "ORDERS",
    masterKeyField: "CustomerId",
    detailForeignKeyField: "CustomerId");

var orderBlock = manager.GetBlock<OrderDto>("ORDERS");
await manager.InsertRecordAsync<OrderDto>("ORDERS", new OrderDto { Status = "Draft" });

Duplicate, export, and import

await manager.DuplicateCurrentRecordAsync("ORDERS");

using var jsonStream = new MemoryStream();
await manager.ExportBlockToJsonAsync("ORDERS", jsonStream);

jsonStream.Position = 0;
await manager.ImportBlockFromJsonAsync("ORDERS_ARCHIVE", jsonStream, clearFirst: true);
Master/detail ownership

Relationship registration, key resolution, and detail synchronization are FormsManager concerns. Keep UnitOfWork focused on entity persistence rather than relationship orchestration.

Key Generation and Record Creation

Key handling is part of the new-record and insert pipeline. FormsManager coordinates the strategy, but the datasource-backed UnitOfWork remains the authority for persisted identities and real sequence refresh.

  1. Preserve explicit keys supplied by caller, default factory, or trigger.
  2. Leave datasource-managed identity keys unset until insert or commit refresh completes.
  3. Prefer datasource or UnitOfWork sequence generation before the in-memory FormsManager sequence provider.
  4. Use FormsManager sequences for deterministic tests or non-database-backed flows.
  5. Handle composite keys field by field.
  6. Do not allocate keys during query, paging, navigation, or cache prefetch.

Explicit key preserved

await manager.InsertRecordAsync<CustomerDto>("CUSTOMERS", new CustomerDto
{
    CustomerId = "ALFKI",
    CompanyName = "Alfreds Futterkiste"
});

Sequence-backed key

manager.SetItemDefault("ORDERS", "Status", () => "Draft");

var nextNumber = manager.GetNextSequence("ORDER_SEQ");

await manager.InsertRecordAsync<OrderDto>("ORDERS", new OrderDto
{
    OrderNumber = nextNumber,
    CustomerId = "ALFKI"
});

Datasource identity key

await manager.InsertRecordAsync<OrderLineDto>("ORDER_LINES", new OrderLineDto
{
    OrderId = existingOrderId,
    ProductId = 12,
    Quantity = 3
    // Leave identity key unset. The datasource refresh path populates it after insert or commit.
});

Default-value factory

manager.SetItemDefault("ORDERS", "CreatedAt", () => DateTime.UtcNow);
manager.SetItemDefault("ORDERS", "CreatedBy", () => Environment.UserName);

await manager.InsertRecordAsync<OrderDto>("ORDERS", new OrderDto());

Triggers, Validation, and LOVs

Trigger system

FormsManager wires trigger behavior across insert, update, delete, query, commit, navigation, key events, and timer expiration. Trigger chaining, dependency tracking, and execution logging are part of the runtime.

Key trigger registration

manager.RegisterKeyTrigger(
    KeyTriggerType.KEY_COMMIT,
    "ORDERS",
    args =>
    {
        Console.WriteLine("Commit key trigger fired");
        return Task.CompletedTask;
    });

Validation

Validation rules can run at item, record, block, form, and cross-block scope. FormsManager integrates validation into field changes, query transitions, navigation, and commit flows.

Validation registration example

manager.Validation.RegisterItemRule("ORDERS", "Quantity", value =>
{
    return value is int quantity && quantity > 0
        ? ValidationResult.Success()
        : ValidationResult.Failure("Quantity must be greater than zero.");
});

LOV orchestration

Register LOV definitions through the LOV helper, then apply a selected row through ShowLOVAsync. That keeps data loading, cache reuse, and related-field population inside the FormsManager orchestration path.

LOV registration and application

var customerLov = LOVDefinition
    .CreateLookup("CUSTOMER_LOV", "Northwind", "Customers", "CustomerId", "CompanyName")
    .MapField("CompanyName", "CustomerName");

manager.LOV.RegisterLOV("ORDERS", "CustomerId", customerLov);

var lovResult = await manager.ShowLOVAsync("ORDERS", "CustomerId", searchText: "alf");
var selected = lovResult.Records.FirstOrDefault();

if (selected != null)
{
    await manager.ShowLOVAsync("ORDERS", "CustomerId", selectedRecord: selected);
}
LOV return-field behavior

LOVManager uses an internal __RETURN_VALUE__ sentinel. Callers should prefer FormsManager.ShowLOVAsync because it translates that sentinel back to the real bound field name before applying the selected values to the current record.

Known Boundaries and Hardening Areas

UI-owned boundaries
  • canvases and tab pages
  • focus rendering and cursor movement between visible controls
  • LOV dialog presentation and row-selection UI
  • keyboard routing from host controls into key-trigger execution
  • status-area and message-area rendering
Residual hardening areas
  • Some CRUD and update seams still rely on reflection-based fallback paths rather than strongly typed UnitOfWork contracts.
  • Programmatic RaiseFormTriggerAsync usage should continue to be reviewed where explicit trigger-raise semantics are expected.
  • Remote cache invalidation remains dependent on the broader host integration and datasource notification model.
