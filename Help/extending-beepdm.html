Extension Overview

BeepDM has two extension patterns depending on whether you are adding support for a new kind of interface that the engine doesn't natively know about:

When to use each pattern

┌──────────────────────────────────────────────────────────────────┐
│ Pattern 1: New IDataSource                                        │
│ Use when: You are adding a new data source type (MySQL,          │
│           REST API, Parquet file, etc.) that the engine should    │
│           query through the standard dm.GetDataSource() flow.     │
│                                                                   │
│ Pattern 2: LoaderExtension for a NEW interface                    │
│ Use when: You are adding a completely new plug-in interface      │
│           that the engine's built-in AssemblyHandler doesn't      │
│           scan for (e.g. IBranch, ICustomWorkflow, IReport).      │
│                                                                   │
│ The difference: IDataSource is a first-class citizen —            │
│ AssemblyHandler already scans for it. Anything else needs a       │
│ custom ILoaderExtention to teach AssemblyHandler to find it.     │
└──────────────────────────────────────────────────────────────────┘

public ConnectionState Openconnection()
{
    // 1. Ensure connection infrastructure exists
    Dataconnection ??= new FileConnection(DMEEditor);
    Dataconnection.ConnectionProp ??= DMEEditor.ConfigEditor
        .DataConnections.FirstOrDefault(c =>
            c.ConnectionName == DatasourceName);

    // 2. Register your format readers (if pluggable)
    FileReaderFactory.RegisterDefaults();

    // 3. Resolve the reader for this file format
    _reader = FileReaderFactory.GetReader(DatasourceType);
    _reader.Initialize(Dataconnection.ConnectionProp);

    // 4. Open the transport layer
    Dataconnection.OpenConnection();
    ConnectionStatus = Dataconnection.ConnectionStatus;

    // 5. Auto-discover schema (populates Entities/EntitiesNames)
    if (ConnectionStatus == ConnectionState.Open)
        InferEntityStructure(EntityName);

    return ConnectionStatus;
}

Step 4: Implement GetEntityStructure() — schema introspection

public EntityStructure GetEntityStructure(string EntityName, bool refresh)
{
    if (!refresh)
    {
        var cached = Entities.FirstOrDefault(e =>
            e.EntityName.Equals(EntityName, StringComparison.OrdinalIgnoreCase));
        if (cached != null) return cached;
    }

    return InferEntityStructure(EntityName);
}

private EntityStructure InferEntityStructure(string entityName)
{
    string filePath = ResolveFilePath(entityName);

    // 1. Read column headers
    string[] headers = _reader.ReadHeaders(filePath);

    // 2. Sample up to 200 rows per column for type inference
    var columnSamples = CollectColumnSamples(filePath, headers, sampleSize: 200);

    // 3. Statistical type inference
    var entity = new EntityStructure { EntityName = entityName };
    foreach (var (colName, sampleValues) in columnSamples)
    {
        var (type, confidence) = TypeInferenceHelper.InferWithStats(sampleValues);
        entity.Fields.Add(new EntityField
        {
            FieldName = colName,
            Fieldtype = type,  // "System.String", "System.Int32", etc.
            FieldCategory = MapToDbFieldCategory(type)
        });
    }

    // 4. Register so dm.GetEntity() can find it
    RegisterEntity(entity);
    return entity;
}

public IEnumerable<object> GetEntity(string EntityName, List<AppFilter> filter)
{
    string filePath = ResolveFilePath(EntityName);
    EntityStructure structure = GetEntityStructure(EntityName, false);
    string[] headers = _reader.ReadHeaders(filePath);

    var headerIndex = BuildHeaderIndex(headers);
    var rows = new List<Dictionary<string, object>>();

    foreach (string[] values in _reader.ReadRows(filePath))
    {
        // Apply filters
        if (!MatchesFilters(filter, structure, values, headerIndex))
            continue;

        // Project columns into a typed dictionary
        var row = ProjectRow(structure, values, headerIndex);
        rows.Add(row);
    }

    return rows;
}

Step 6: Implement CRUD (InsertEntity)

public IErrorsInfo InsertEntity(string EntityName, object InsertedData)
{
    var entity = GetEntityStructure(EntityName, false);
    string filePath = ResolveFilePath(EntityName);
    string[] headers = GetOrCreateHeaders(filePath, entity);

    // Convert object → ordered row matching entity.Fields
    var rowValues = ToOrderedArray(InsertedData, entity);

    // Validate before writing
    var validation = ValidateRow(entity, rowValues, headers, rowIndex: -1, "INSERT");
    if (!ShouldWriteRow(validation))
    {
        ErrorObject.Flag = Errors.Failed;
        ErrorObject.Message = validation.ErrorMessage;
        return ErrorObject;
    }

    _reader.AppendRow(filePath, entity.Fields, rowValues);
    ErrorObject.Flag = Errors.Ok;
    return ErrorObject;
}

How AssemblyHandler Discovers It

The complete chain from DLL to running data source:

1. Compile your DLL (e.g. MyPlugin.dll)
            │
2. AssemblyHandler.LoadAllAssembly() scans the DLL
            │
3. IScanningService finds classes with [AddinAttribute] implementing IDataSource
            │
4. Creates AssemblyClassDefinition in ConfigEditor.DataSourcesClasses
   - className = "FileDataSource"
   - type = typeof(FileDataSource)
   - PackageName = "TheTechIdea.Beep.FileManager"
            │
5. User calls dm.GetDataSource("myfile.csv")
            │
6. DMEEditor links ConnectionProperties → ConnectionDriversConfig
   → classHandler = "FileDataSource"
            │
7. DMEEditor matches classHandler to DataSourcesClasses by className
   → finds AssemblyClassDefinition with type = typeof(FileDataSource)
            │
8. Finds constructor (string, IDMLogger, IDMEEditor, DataSourceType, IErrorsInfo)
            │
9. Instantiates via ObjectActivator<IDataSource>
   → FileDataSource instance is ready
            │
10. dm.GetDataSource() returns it. Consumer calls ds.Openconnection()

Pattern 2: Creating a Loader Extension for a New Interface

Why You Need a Loader Extension

AssemblyHandler has built-in scanning for a fixed set of interfaces: IDataSource, IDM_Addin, IWorkFlowAction, IRuleParser, IDefaultValueResolver, IFileFormatReader, and a few others.

If you create a brand-new interface (e.g. IBranch, IReport, ICustomDashboard) and want the engine to discover implementations of it, you must write an ILoaderExtention that teaches the engine how to find them.

ILoaderExtention Interface

// DataManagementModelsStandard/Tools/ILoaderExtention.cs
public interface ILoaderExtention
{
    IErrorsInfo LoadAllAssembly();              // Scan ALL loaded assemblies
    IErrorsInfo Scan(assemblies_rep assembly);  // Scan a single assembly reference
    IErrorsInfo Scan(Assembly assembly);         // Scan a single System.Reflection.Assembly
    IErrorsInfo Scan();                          // No-arg convenience scan
}

Step 3: Implement ILoaderExtention — the scanner

// Beep.Branchs/TheTechIdea.Beep.Vis.Loader/BranchLoaderExtension.cs
namespace AssemblyLoaderExtension
{
    public class BranchLoaderExtension : ILoaderExtention
    {
        public AppDomain CurrentDomain { get; set; }
        public IAssemblyHandler Loader { get; set; }

        public BranchLoaderExtension(IAssemblyHandler ploader)
        {
            Loader = ploader;
            CurrentDomain = AppDomain.CurrentDomain;
            // Wire up assembly resolution for dependency loading
            CurrentDomain.AssemblyResolve += Loader.CurrentDomain_AssemblyResolve;
        }

        // === Scan ALL loaded assemblies ===
        public IErrorsInfo LoadAllAssembly()
        {
            var er = new ErrorsInfo();
            foreach (var item in Loader.Assemblies)
            {
                try { ScanAssembly(item.DllLib); }
                catch (Exception ex) { /* log and continue */ }
            }
            return er;
        }

        // === The core scanner — detects IBranch implementations ===
        private bool ScanAssembly(Assembly asm)
        {
            Type[] types;
            try
            {
                types = asm.GetTypes();   // Try all types first
                if (types == null)
                    types = asm.GetExportedTypes();  // Fall back to public only
            }
            catch { return false; }

            if (types == null) return false;

            foreach (var type in types)
            {
                var typeInfo = type.GetTypeInfo();

                // ★ THIS IS THE KEY LINE ★
                // Check if the type implements YOUR interface
                if (typeInfo.ImplementedInterfaces.Contains(typeof(IBranch)))
                {
                    // Register it using AssemblyHandler's helper
                    var classDef = Loader.GetAssemblyClassDefinition(
                        typeInfo, "IBranch");

                    // Add to the dedicated collection in ConfigEditor
                    Loader.ConfigEditor.BranchesClasses.Add(classDef);
                }

                // You can scan for MULTIPLE interfaces in one extension:
                // if (typeInfo.ImplementedInterfaces.Contains(typeof(IReport)))
                //     Loader.ConfigEditor.ReportClasses.Add(...);
            }

            return true;
        }

        // === Convenience overloads ===
        public IErrorsInfo Scan(assemblies_rep assembly)
        {
            try { ScanAssembly(assembly.DllLib); }
            catch (Exception ex) { /* ... */ }
            return new ErrorsInfo { Flag = Errors.Ok };
        }

        public IErrorsInfo Scan(Assembly assembly)
        {
            try { ScanAssembly(assembly); }
            catch (Exception ex) { /* ... */ }
            return new ErrorsInfo { Flag = Errors.Ok };
        }

        public IErrorsInfo Scan()
        {
            LoadAllAssembly();
            return new ErrorsInfo { Flag = Errors.Ok };
        }
    }
}

Step 4: Auto-Discovery — No Manual Registration Needed

You do NOT need to manually register the extension. AssemblyHandler (and SharedContextAssemblyHandler) automatically scan for ILoaderExtention implementations during LoadAllAssembly().

The auto-discovery flow in AssemblyHandler.Scanning.cs:99-300:

How AssemblyHandler discovers and runs ILoaderExtentions automatically

// Phase 1: Discovery (AssemblyHandler.Scanning.cs line 104)
// During the main assembly scan, every type is checked:
private void ProcessTypeInfo(TypeInfo typeInfo, Assembly asm)
{
    if (typeInfo.ImplementedInterfaces.Contains(typeof(ILoaderExtention)))
    {
        LoaderExtensions.Add(typeInfo);    // Store the Type
        LoaderExtensionClasses.Add(
            GetAssemblyClassDefinition(typeInfo, "ILoaderExtention"));
    }
    // ... also checks IDataSource, IDM_Addin, IRuleParser, etc.
}

// Phase 2: Auto-Instantiation & Auto-Scan (line 256)
private void ScanExtensions()
{
    foreach (Type item in LoaderExtensions)
    {
        // 1. Instantiate automatically — passes 'this' (IAssemblyHandler)
        ILoaderExtention cls = (ILoaderExtention)
            Activator.CreateInstance(item, new object[] { this });

        // 2. Store the instance
        LoaderExtensionInstances.Add(cls);

        // 3. Auto-scan EVERY loaded assembly through the extension
        foreach (Assembly assembly1 in LoadedAssemblies)
        {
            if (!assembly1.FullName.StartsWith("System") &&
                !assembly1.FullName.StartsWith("Microsoft"))
            {
                cls.Scan(assembly1);  // ← Your ScanAssembly() gets called
            }
        }
    }
}
All you need to do:
  1. Implement ILoaderExtention
  2. Provide a constructor that takes IAssemblyHandler
  3. Place the compiled DLL in LoadingExtensions/ or ProjectClasses/
  4. LoadAllAssembly() does the rest — discovers, instantiates, and calls Scan()

The constructor signature matters. AssemblyHandler instantiates your extension with new BranchLoaderExtension(this) — so your constructor must accept an IAssemblyHandler:

    Loader = ploader;   // ← Now you have access to assemblies, ConfigEditor, etc.

Pattern 3: Consuming the Discovered Types

BeepAppTree.CreateRootTree() — Building a UI from IBranch

Source: Beep.Winform/.../BeepTreeView/BeepAppTree.cs (line 413)

Once the BranchLoaderExtension has populated BranchesClasses, the UI layer reads those definitions, instantiates each branch via reflection, and builds the tree. Here's the complete flow:

The full flow — from AssemblyClassDefinition to visible tree node

public IErrorsInfo CreateRootTree()
{
    // === PHASE 1: Build Genre (category) nodes ===
    // Genre branches group related branches under a category header
    foreach (AssemblyClassDefinition def in DMEEditor.ConfigEditor.BranchesClasses
        .Where(b => b.VisSchema?.BranchType == EnumPointType.Genre)
        .OrderBy(b => b.Order))
    {
        // a) Resolve the Type from the fully-qualified name
        Type type = DMEEditor.assemblyHandler.GetType(def.PackageName);
        // def.PackageName = "BeepEnterprize.Vis.Module.MyGenreBranch"

        // b) Find a parameterless constructor
        ConstructorInfo ctor = type.GetConstructors()
            .FirstOrDefault(c => c.GetParameters().Length == 0);

        // c) Create an activator (compiled expression tree, fast)
        ObjectActivator<IBranch> activator = GetActivator<IBranch>(ctor);

        // d) Instantiate the branch
        IBranch genre = activator();

        // e) Configure it
        genre.Name = def.PackageName;
        genre.ID = nextId++;
        genre.BranchID = genre.ID;

        // f) Create a UI node in the tree
        CreateNode(genre.ID, genre);
        Branches.Add(genre);
    }

    // === PHASE 2: Build Root branches ===
    // Root branches are top-level navigation items
    foreach (AssemblyClassDefinition def in DMEEditor.ConfigEditor.BranchesClasses
        .Where(b => b.VisSchema?.BranchType == EnumPointType.Root)
        .OrderBy(b => b.Order))
    {
        // Same reflection pattern: resolve type → find constructor → activate
        Type type = DMEEditor.assemblyHandler.GetType(def.PackageName);
        ConstructorInfo ctor = type.GetConstructors()
            .FirstOrDefault(c => c.GetParameters().Length == 0);

        ObjectActivator<IBranch> activator = GetActivator<IBranch>(ctor);
        IBranch branch = activator();

        // Configure branch
        branch.Name = def.PackageName;
        branch.ID = nextId++;
        branch.DMEEditor = DMEEditor;
        branch.TreeEditor = this;   // branch can now interact with the tree

        // === PHASE 3: Route branch to the right Genre ===
        // Match branch to its parent Genre by menu category
        var parentGenre = GenreBranches.FirstOrDefault(g =>
            g.Item2 == def.classProperties.menu);

        if (parentGenre != null)
        {
            // Add under the Genre category node
            AddBranch(parentGenre.Item1, branch);
        }
        else
        {
            // Top-level root node
            CreateNode(branch.ID, branch);
        }
    }
}

// === AddBranch: recursive child construction ===
public IErrorsInfo AddBranch(IBranch parent, IBranch child)
{
    // 1. Link parent-child
    child.ParentBranch = parent;
    child.ParentBranchID = parent.ID;

    // 2. Create UI node under parent
    SimpleItem node = CreateNode(child.ID, child);
    AddNodeWithBranch(node, parent.GuidID);

    // 3. Give the branch a chance to populate its children
    child.SetConfig(this, DMEEditor, parent,
        child.BranchText, child.ID,
        child.BranchType, null);

    // 4. Recursively build child branches
    parent.ChildBranchs.Add(child);
    child.CreateChildNodes();  // ← Branch's own logic to build sub-items

    // 5. Recurse into grandchildren
    foreach (var grandchild in child.ChildBranchs)
        AddBranch(child, grandchild);

    return DMEEditor.ErrorObject;
}

The three branch types used in the tree

public enum EnumPointType
{
    Genre,   // Category header — groups related branches
             //   e.g. "Data Sources", "Reports", "Administration"
    Root,    // Top-level navigation item
             //   e.g. "Northwind DB", "Sales Dashboard"
    Child    // Nested item under a Root or another Child
             //   e.g. "Customers" table, "Monthly Report"
}

// Each branch type gets handled differently:
// Genre  → Created first as top-level category containers
// Root   → Added under a Genre, or standalone at root
// Child  → Created by the branch's own CreateChildNodes()

What a concrete IBranch implementation looks like

// A branch that represents a database connection in the tree
[AddinAttribute(Name = "DatabaseBranch",
    Caption = "Database Connection",
    Category = DatasourceCategory.RDBMS)]
public class DatabaseBranch : IBranch
{
    // IBranchID properties
    public string Name { get; set; }
    public string BranchText { get; set; } = "Database";
    public EnumPointType BranchType { get; set; } = EnumPointType.Root;
    public int BranchID { get; set; }
    public string IconImageName { get; set; } = "database.svg";

    // IBranch properties
    public int ID { get; set; }
    public IDMEEditor DMEEditor { get; set; }
    public IDataSource DataSource { get; set; }
    public List<IBranch> ChildBranchs { get; set; } = new();

    // CreateChildNodes — called by the tree to build the subtree
    public IErrorsInfo CreateChildNodes()
    {
        if (DataSource == null) return DMEEditor.ErrorObject;

        // Discover tables from the connected database
        foreach (var entityName in DataSource.EntitiesNames)
        {
            var child = new TableBranch
            {
                Name = entityName,
                BranchText = entityName,
                BranchType = EnumPointType.Child,
                IconImageName = "table.svg",
                DataSource = DataSource,
                DMEEditor = DMEEditor
            };
            ChildBranchs.Add(child);
        }

        return DMEEditor.ErrorObject;
    }

    public IErrorsInfo SetConfig(ITree tree, IDMEEditor editor,
        IBranch parent, string text, int id,
        EnumPointType branchType, string imageName)
    {
        TreeEditor = tree;
        DMEEditor = editor;
        // Open the database connection
        DataSource = DMEEditor.GetDataSource(Name);
        DataSource?.Openconnection();
        return DMEEditor.ErrorObject;
    }
}

Why it works automatically

AssemblyHandler.ScanExtensions() (line 256 in AssemblyHandler.Scanning.cs) iterates over all discovered ILoaderExtention types, instantiates them with new YourExtension(this), and calls Scan(assembly) for every non-System assembly. The only requirement: your DLL must be in a scanned directory and your constructor must accept IAssemblyHandler.
