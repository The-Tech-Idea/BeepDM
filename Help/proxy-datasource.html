Overview

ProxyDataSource wraps one or more concrete IDataSource instances behind a single, transparent facade that the rest of the application treats as an ordinary data source. It adds the following capabilities without any changes to calling code:

Smart Routing

Four strategies: Weighted Latency, Least Outstanding Requests, Round Robin, Health Weighted. Reads can be load-balanced across replicas; writes are pinned to the Primary role.

Circuit Breaker

Per-source three-state (Closed / HalfOpen / Open) breaker with severity-weighted failure accumulation and configurable reset windows.

Retry & Failover

Automatic retry with exponential back-off and jitter. Seamless failover across candidates on transient errors. Non-idempotent writes execute exactly once.

Connection Pool

Per-source bounded pool (MaxPoolSize=10). Idle connections expire after ConnectionTimeout (5 min). TTL-based eviction runs on the health-check timer.

Caching

Stale-while-revalidate entity cache backed by CacheScope. Write-through invalidation keeps cache consistent on every insert/update/delete.

Watchdog

Tight-cycle background loop probes the write-primary independently of the health-check timer. Promotes a replica automatically when the primary fails, demotes it when the primary recovers.

Observability

Per-source DataSourceMetrics counters and a rolling 500-sample latency buffer that yields p50/p95/p99 SLO snapshots on demand.

Audit Trail

Every proxied operation writes a ProxyAuditEntry to a swappable IProxyAuditSink. Ships with a file-based rolling JSON-lines sink out of the box.

Drop-in replacement: ProxyDataSource implements both IProxyDataSource and IDataSource, so you can pass it anywhere a normal data source is expected. All resilience features are invisible to the caller.

Request lifecycle

caller → ProxyDataSource.GetEntity(...)
            │
            ├─ SelectCandidates()          [Routing]
            │     filter healthy + closed circuits
            │     order by routing strategy
            │
            ├─ for each candidate:
            │     GetPooledConnection()
            │     for 1..MaxRetries:
            │         execute operation
            │         ├─ success → RecordSuccess, emit audit, return
            │         └─ failure → ProxyErrorClassifier.Classify()
            │                       ├─ non-retryable → throw immediately
            │                       ├─ saturation    → 2× back-off, retry
            │                       └─ transient     → exp back-off+jitter, retry
            │
            └─ all candidates exhausted → log + throw last exception

    int?         healthCheckIntervalMs   = null)

Wraps the scalar parameters into a ProxyPolicy internally. Provided for compatibility with older call sites; prefer the policy overload for new code.

Both constructors call InitializeFromPolicy() which starts the health-check timer, seeds per-source circuit breakers, and initialises the cache provider if enabled.

ProxyPolicy

ProxyPolicy is the single source of truth for every tunable behaviour. It is applied at construction time and can be swapped at runtime via ApplyPolicy(ProxyPolicy) without recreating the proxy.

Top-level properties

PropertyTypeDefaultDescription
Namestring"Default"Human-readable policy name used in log messages
Versionint1Monotonic version for change tracking
RoutingStrategyProxyRoutingStrategyWeightedLatencyAlgorithm used to order candidate sources for each request
HealthCheckIntervalMsint30 000Interval between background health checks (ms)
ResilienceProxyResilienceProfileBalanced presetRetry / circuit-breaker parameters
CacheProxyCacheProfiledisabledEntity cache configuration

Runtime hot-swap

// Change to aggressive failover without recreating the proxy
var aggressivePolicy = new ProxyPolicy
{
    Name            = "HighAvailability",
    RoutingStrategy = ProxyRoutingStrategy.HealthWeighted,
    Resilience      = ProxyResilienceProfile.AggressiveFailover,
    HealthCheckIntervalMs = 10_000
};

proxy.ApplyPolicy(aggressivePolicy);   // thread-safe; stops old timer, reinitialises circuits
ApplyPolicy stops the old health-check timer before swapping the policy reference to prevent a duplicate timer leak. Always use ApplyPolicy rather than mutating _policy directly.

Routing Strategies

On every read operation, SelectCandidates() builds an ordered list of healthy (non-open-circuit) sources and passes it to the active routing algorithm. The execution loop tries candidates in that order with retries. Writes are pre-filtered to Primary-role sources only.

StrategyAlgorithmBest for
WeightedLatency Sort by weight descending, then average latency ascending Mixed-performance backends where weight and latency both matter
RoundRobin Monotonic counter mod candidate count; rotates list so selected index is first Homogeneous backends, simple equal distribution
LeastOutstandingRequests Sort by number of in-flight connections ascending Backends with variable concurrency limits or connection pools
HealthWeighted Weight = 1 / (failure rate + ε); source with lowest failure rate gets most traffic Sources that flip between healthy and degraded frequently
When all candidates have open circuits (fully degraded state), the proxy falls back to including all registered sources regardless of circuit state to avoid total unavailability.

Circuit Breaker

Each registered datasource gets its own CircuitBreaker instance managed by the active ICircuitStateStore. The breaker uses three states:

Closed
Normal operation
⟶ failures ≥ threshold ⟶
Open
All calls rejected
⟶ reset timeout elapsed ⟶
Half-Open
Probe calls allowed
⟶ successes ≥ threshold → Closed
⟵ failure → Open

Severity-weighted failure accumulation

SeverityWeightExample exceptions
Low1OperationCanceledException, generic connection errors
Medium1TimeoutException, deadlock, transient errors
High2UnauthorizedAccessException, persistent errors
CriticalImmediate openOutOfMemoryException (saturation)

Swappable state store

The ICircuitStateStore interface allows replacing the default in-process store with a distributed backend so circuit state is shared across multiple proxy instances: 

public interface ICircuitStateStore
{
    bool CanExecute(string dsName);
    void RecordSuccess(string dsName);
    void RecordFailure(string dsName, ProxyErrorSeverity severity = ProxyErrorSeverity.Medium);
    void Initialize(string dsName, int failureThreshold, TimeSpan resetTimeout, int successThreshold = 2);
    void Remove(string dsName);
}

Use the provided ProxyRedisCircuitStateStore for clusters that share state across processes, or implement the interface against any storage backend.

Connection Pool

Each registered datasource has its own bounded ConcurrentQueue<PooledConnection>. Connections are leased from the pool before each operation and returned after. Idle eviction runs on the health-check timer interval.

KnobDefaultDescription
MaxPoolSize10Maximum number of connections per datasource
ConnectionTimeout5 minIdle threshold; connections older than this are closed on next eviction pass

Pool API

// Internal — used by execution helpers
IDataSource lease  = proxy.GetPooledConnection(dsName);   // borrow from pool (or open new)
proxy.ReturnConnection(dsName, lease);                    // return lease after use
IDataSource direct = proxy.GetConnection(dsName);         // bypass pool — new connection always

Caching

Enabled via ProxyCacheProfile inside ProxyPolicy. The cache is backed by a CacheScope from the BeepDM caching subsystem. The proxy uses a stale-while-revalidate strategy: a soft TTL (the visible expiry) and a hard TTL (stale extension window). When a key is between soft and hard TTL the stale value is returned immediately while a background refresh populates the cache.

PropertyDefaultDescription
EnabledfalseMust be explicitly set to true to activate the cache
DefaultExpiration—Soft TTL for cached entries
StaleWindow—Additional window during which stale data is served while revalidating
MaxItems—Upper bound on cached entries before LRU eviction
TierEntityProfileCache provider type (InMemory by default)
Consistency—WriteThrough triggers automatic invalidation on write operations

Cache API

// Read with cache (respects stale-while-revalidate)
var rows = proxy.GetEntityWithCache("Orders", filters, expiration: TimeSpan.FromMinutes(5));

// Manual invalidation
proxy.InvalidateCache("Orders");    // invalidate one entity
proxy.InvalidateCache();            // clear all cached entries
When CacheConsistency = WriteThrough any call to InsertEntity, UpdateEntity, or DeleteEntity automatically calls InvalidateCache(entityName) before returning.

Watchdog

The watchdog runs as a background Task on a tight cycle (default 5 s) and probes every registered datasource with a lightweight liveness call capped at WatchdogProbeTimeoutMs (default 2 s). It maintains independent consecutive failure/success counters per source, separate from the main circuit breaker counters.

Role management

  1. When a Primary source fails WatchdogFailureThreshold consecutive probes, the watchdog promotes the highest-weighted healthy Replica to Primary.
  2. The promotion is recorded in _originalRoles so the role can be reversed.
  3. When the original Primary recovers (WatchdogRecoveryThreshold consecutive successes), the watchdog demotes it back to Replica and restores the promoted Replica.

Knobs & events

MemberDefaultDescription
WatchdogIntervalMs5 000Probe loop delay (ms)
WatchdogProbeTimeoutMs2 000Max ms for a single probe call
WatchdogFailureThreshold2Consecutive failures before promotion
WatchdogRecoveryThreshold3Consecutive successes before demotion
OnRolePromoted—Fired when a Replica is promoted to Primary
OnRoleDemoted—Fired when the original Primary is restored

Usage

proxy.WatchdogIntervalMs       = 3_000;  // tighter probe on critical DB
proxy.WatchdogFailureThreshold = 3;

proxy.OnRolePromoted += (_, e) =>
    Console.WriteLine($"{e.DataSourceName} promoted to {e.NewRole}");

proxy.StartWatchdog();

// ... application lifetime ...

proxy.StopWatchdog();  // graceful — waits up to 5 s for loop exit

Observability & SLO

The observability layer maintains a bounded 500-sample circular latency buffer per datasource and exposes p50/p95/p99 percentile summaries through the SLO snapshot API. Samples are recorded automatically by the execution helpers on every successful call.

SLO snapshot API

// Single source snapshot
ProxySloSnapshot snap = proxy.GetSloSnapshot("orders-primary");
Console.WriteLine($"p99={snap.P99LatencyMs:F1}ms  err={snap.ErrorRatePercent:F2}%  cache={snap.CacheHitRatio:P}");

// All sources at once
IReadOnlyList<ProxySloSnapshot> all = proxy.GetAllSloSnapshots();
foreach (var s in all)
    Console.WriteLine($"{s.DataSourceName}: p95={s.P95LatencyMs:F1}ms");

Raw metrics access

IDictionary<string, DataSourceMetrics> metrics = proxy.GetMetrics();
foreach (var (name, m) in metrics)
    Console.WriteLine($"{name}: total={m.TotalRequests}  failed={m.FailedRequests}  breaks={m.CircuitBreaks}");

Every proxied operation emits a ProxyAuditEntry to the configured IProxyAuditSink after the operation completes — whether it succeeded or failed.

Provided sinks

ClassDescription
NullProxyAuditSinkSingleton no-op — default when no sink is supplied. Zero allocations.
FileProxyAuditSinkRolling daily JSON-lines file: {dir}/proxy-audit-YYYYMMDD.jsonl. Uses a bounded background queue so the data path is never blocked.
 
// Enable file audit sink at construction
var sink = new FileProxyAuditSink(directory: @"C:\Logs\ProxyAudit", queueCapacity: 10_000);

var proxy = new ProxyDataSource(
    dmeEditor       : editor,
    dataSourceNames : new List<string> { "db1", "db2" },
    auditSink       : sink);

// ... use proxy ...

sink.Dispose();   // flushes background queue, waits up to 5 s
Implement IProxyAuditSink to write entries to a database, telemetry provider (Application Insights, OpenTelemetry), or any other destination. The sink's Write method must not throw and must complete quickly.

Error Taxonomy

ProxyErrorClassifier.Classify(Exception) maps every exception to a category + severity pair that drives retry eligibility and circuit-breaker weight:

CategoryRetry?SeverityTrigger
Transient✅ YesLow–MediumOperationCanceledException, IOException, "connection" in message
Timeout✅ YesMediumTimeoutException, "timeout" in message
AuthFailure❌ NoHighUnauthorizedAccessException
Saturation⏳ Back-off 2×CriticalOutOfMemoryException, "connection pool full"
Persistent❌ NoHighSchema errors, bad SQL, all other exceptions
Unknown❌ NoMediumUnmatched exceptions

Saturation is special: the proxy backs off with a delay proportional to attempt × 2 (double the normal back-off) to reduce load on an overwhelmed backend rather than hammering it with rapid retries.

ProxyCluster

ProxyCluster implements IProxyCluster : IProxyDataSource and represents a cluster of ProxyNode instances behind a single IDataSource facade. Use it when you need a second tier of abstraction: clusters of proxies rather than clusters of bare datasources.

Node management

var cluster = new ProxyCluster(
    clusterPolicy : new ProxyPolicy { RoutingStrategy = ProxyRoutingStrategy.LeastOutstandingRequests },
    auditSink     : new FileProxyAuditSink(@"C:\Logs\Cluster"));

// Add nodes
cluster.AddNode(new ProxyNode("primary",  primaryProxy,  weight: 2, role: ProxyDataSourceRole.Primary));
cluster.AddNode(new ProxyNode("replica1", replicaProxy1, weight: 1, role: ProxyDataSourceRole.Replica));
cluster.AddNode(new ProxyNode("replica2", replicaProxy2, weight: 1, role: ProxyDataSourceRole.Replica));

// Remove a node gracefully (drains in-flight requests)
cluster.RemoveNode("replica2");

// Inspect nodes
IReadOnlyList<IProxyNode> nodes = cluster.GetNodes();
foreach (var n in nodes)
    Console.WriteLine($"{n.NodeId}: alive={n.IsAlive} weight={n.Weight} inflight={n.InFlightCount}");

Usage Examples

Example 1 — Basic read with default policy

var proxy = new ProxyDataSource(
    dmeEditor       : editor,
    dataSourceNames : new List<string> { "orders-primary", "orders-replica" });

proxy.Openconnection();

var rows = proxy.GetEntity("Orders", new List<AppFilter>
{
    new AppFilter { FieldName = "Status", Operator = "=", FilterValue = "Open" }
});

foreach (var row in rows)
    Console.WriteLine(row);

proxy.Closeconnection();
proxy.Dispose();

Example 2 — Custom policy with caching

var policy = new ProxyPolicy
{
    Name            = "ProductsPolicy",
    RoutingStrategy = ProxyRoutingStrategy.WeightedLatency,
    Resilience = new ProxyResilienceProfile
    {
        ProfileType         = ProxyResilienceProfileType.Custom,
        MaxRetries          = 3,
        RetryBaseDelayMs    = 200,
        FailureThreshold    = 5,
        CircuitResetTimeout = TimeSpan.FromSeconds(30)
    },
    Cache = new ProxyCacheProfile
    {
        Enabled           = true,
        DefaultExpiration = TimeSpan.FromSeconds(60),
        Consistency       = ProxyCacheConsistency.WriteThrough
    }
};

var proxy = new ProxyDataSource(
    dmeEditor       : editor,
    dataSourceNames : new List<string> { "products-db1", "products-db2" },
    policy          : policy);

proxy.Openconnection();
var result = proxy.RunQuery("SELECT COUNT(*) FROM Products");
proxy.Dispose();

Example 3 — Read/write role separation

var proxy = new ProxyDataSource(
    dmeEditor       : editor,
    dataSourceNames : new List<string> { "sales-primary", "sales-replica-1", "sales-replica-2" });

// Assign roles
proxy.SetRole("sales-primary",   ProxyDataSourceRole.Primary);
proxy.SetRole("sales-replica-1", ProxyDataSourceRole.Replica);
proxy.SetRole("sales-replica-2", ProxyDataSourceRole.Replica);

proxy.Openconnection();

// Read — may land on any healthy node
var items = proxy.GetEntity("SaleItems", null);

// Write — guaranteed to go to Primary-role source only
proxy.InsertEntity("Orders", new { OrderId = Guid.NewGuid(), Total = 99.99 });

proxy.Dispose();

Example 4 — Watchdog with role promotion events

var proxy = new ProxyDataSource(
    dmeEditor       : editor,
    dataSourceNames : new List<string> { "db-primary", "db-replica" });

proxy.SetRole("db-primary", ProxyDataSourceRole.Primary);
proxy.SetRole("db-replica", ProxyDataSourceRole.Replica);

proxy.WatchdogIntervalMs       = 3_000;
proxy.WatchdogFailureThreshold = 2;
proxy.WatchdogRecoveryThreshold = 4;

proxy.OnRolePromoted += (_, e) =>
    logger.LogWarning($"PROMOTED: {e.DataSourceName} is now {e.NewRole}");
proxy.OnRoleDemoted += (_, e) =>
    logger.LogInformation($"DEMOTED: {e.DataSourceName} restored to {e.NewRole}");

proxy.Openconnection();
proxy.StartWatchdog();

// ... application lifetime ...

proxy.StopWatchdog();
proxy.Dispose();

Example 5 — Audit + observability

using var sink = new FileProxyAuditSink(@"C:\Logs\ProxyAudit");

var proxy = new ProxyDataSource(
    dmeEditor       : editor,
    dataSourceNames : new List<string> { "analytics-db1", "analytics-db2" },
    auditSink       : sink);

proxy.Openconnection();

// ... run operations ...

// Check SLO
foreach (var snap in proxy.GetAllSloSnapshots())
{
    Console.WriteLine(
        $"{snap.DataSourceName}: p50={snap.P50LatencyMs:F1}ms  " +
        $"p99={snap.P99LatencyMs:F1}ms  " +
        $"err={snap.ErrorRatePercent:F2}%  " +
        $"cache={snap.CacheHitRatio:P}");
}

proxy.Dispose();
// sink.Dispose() called by using — flushes queue

Example 6 — ProxyCluster

// Build individual proxies (one per physical server)
var primary = new ProxyDataSource(editor, new List<string> { "server-a" });
var replica = new ProxyDataSource(editor, new List<string> { "server-b" });

// Compose into a cluster
var cluster = new ProxyCluster(
    clusterPolicy : new ProxyPolicy { RoutingStrategy = ProxyRoutingStrategy.LeastOutstandingRequests });

cluster.AddNode(new ProxyNode("primary",  primary,  weight: 2, role: ProxyDataSourceRole.Primary));
cluster.AddNode(new ProxyNode("replica1", replica,  weight: 1, role: ProxyDataSourceRole.Replica));

cluster.OnNodeDown      += (_, e) => Console.WriteLine($"Node down: {e.NodeId}");
cluster.OnNodeRestored  += (_, e) => Console.WriteLine($"Node up:   {e.NodeId}");

// Use the cluster exactly like a plain IDataSource
IDataSource ds = cluster;
var data = ds.GetEntity("Products", null);

Example 7 — Hot policy swap

// Start with balanced policy
var proxy = new ProxyDataSource(editor, new List<string> { "db1", "db2" });

// During an incident: switch to aggressive failover in place
proxy.ApplyPolicy(new ProxyPolicy
{
    Name            = "Incident-2024",
    RoutingStrategy = ProxyRoutingStrategy.HealthWeighted,
    Resilience      = ProxyResilienceProfile.AggressiveFailover,
    HealthCheckIntervalMs = 5_000
});

// After the incident: restore normal policy
proxy.ApplyPolicy(new ProxyPolicy { Resilience = ProxyResilienceProfile.Balanced });
