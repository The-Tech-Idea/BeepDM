Overview

BeepSyncManager is the modern replacement for the legacy DataSyncManager. It is a partial class split across nine focused files and orchestrates one-way or bidirectional data synchronization between any two registered IDataSource instances.

Key design principles

PrincipleHow it is applied
Single ResponsibilityEach partial file / helper class owns one domain (CDC, conflict, preflight, …)
Async-firstAll long-running operations are async Task with CancellationToken
Rule-engine integrationEvery policy gate (DQ, conflict, retry, promotion) can be overridden by named rules
TestabilityAll helpers are injected via interfaces; the manager accepts mocks transparently
Progress & observabilityIProgress<PassedArgs> propagated at every step; SyncMetrics emitted per run

Quick Start

Minimal example — one-way full sync of a Customers table.

// 1. Create the manager
var syncManager = new BeepSyncManager(dmeEditor);

// 2. Describe the sync
var schema = new DataSyncSchema
{
    Id                        = Guid.NewGuid().ToString(),
    SourceDataSourceName      = "SourceDB",
    DestinationDataSourceName = "DestDB",
    SourceEntityName          = "Customers",
    DestinationEntityName     = "Customers",
    SourceKeyField            = "CustomerId",
    DestinationKeyField       = "CustomerId",
    SyncType                  = "Full",       // "Full" | "Incremental"
    SyncDirection             = "OneWay",     // "OneWay" | "Bidirectional"
    BatchSize                 = 500,
};

// 3. Auto-map fields (name matching)
var fieldHelper = new FieldMappingHelper(dmeEditor);
foreach (var m in fieldHelper.AutoMapFields("SourceDB", "Customers", "DestDB", "Customers"))
    schema.MappedFields.Add(m);

// 4. Register & run
syncManager.AddSyncSchema(schema);
var progress = new Progress<PassedArgs>(p => Console.WriteLine(p.Messege));
await syncManager.SyncDataAsync(schema, CancellationToken.None, progress);

// 5. Inspect result
Console.WriteLine($"Status: {schema.SyncStatus}  LastSync: {schema.LastSyncDate:u}");

// 6. Persist
await syncManager.SaveSchemasAsync();

SyncDataAsync — Full Orchestration

Signature

Task SyncDataAsync(
    DataSyncSchema         schema,
    CancellationToken      token          = default,
    IProgress<PassedArgs> progress       = null,
    IImportErrorStore      errorStore     = null,
    IImportHistoryStore    historyStore   = null
);

Pipeline steps

When called, SyncDataAsync executes the following ordered pipeline:

#StepWhat happens
1Structural validationValidateSchema — null, missing names, empty mappings
2PreflightRunPreflightAsync — connectivity, rule checks, defaults readiness
3DQ mapping quality gateRule sync.dq.mapping-quality inspects field map completeness (if rules enabled)
4Rule audit telemetryRule sync.audit.pre-run emits tracing metadata
5Performance profileRule sync.perf.profile selects RU/batch tuning values
6Retry / checkpoint loadLoads existing checkpoint from ActiveCheckpoint; classifies last error if any
7CDC filter buildBuildCdcFilterContext — calculates watermark window, filters, late-arrival rule
8ImportDelegates to DataImportManager.RunImportAsync with the CDC-enriched import config
9Bidirectional reverse importIf SyncDirection = "Bidirectional", swaps source ↔ dest and re-imports
10Conflict gateTryEvaluateConflictGate runs built-in or rule-driven resolution; quarantines if required
Cancellation — Pass a CancellationToken from a UI button or scheduler. The sync manager propagates it to RunImportAsync and all awaited helpers. On cancellation the schema SyncStatus is set to "Cancelled".

Bulk Sync

When you have multiple schemas registered, the bulk helpers iterate them in one call.

Sequential

// Async — processes schemas one after another
await syncManager.SyncAllDataAsync(CancellationToken.None, progress);

// Sync fallback
syncManager.SyncAllData();

Parallel

// Runs schemas in parallel, bounded by SyncPerformanceProfile.MaxDegreeOfParallelism
await syncManager.SyncAllDataParallelAsync(CancellationToken.None, progress);

The semaphore count is read from the schema's performance profile rule result (default 4). This avoids overwhelming shared data sources under load.

Preflight Validation

Method

Task<SyncPreflightReport> RunPreflightAsync(
    DataSyncSchema    schema,
    CancellationToken token = default
);

Checks performed

#CheckGate type
1Structural validation — null fields, missing source/dest namesHard fail
2Rule Engine gate — sync.plan.validate (if rules enabled)Soft / configurable
3Defaults profile readiness — IDefaultsManager.IsReadyWarning

SyncPreflightReport

var report = await syncManager.RunPreflightAsync(schema);

if (!report.IsApproved)
{
    foreach (var issue in report.Issues)
        Console.WriteLine($"[{issue.Severity}] {issue.Code}: {issue.Message}");
    return; // block the run
}

Console.WriteLine($"Rules passed : {report.RulesPassed}");
Console.WriteLine($"Defaults ready: {report.DefaultsReady}");

CDC & Watermark (Incremental Sync)

Set SyncType = "Incremental" and configure WatermarkPolicy to restrict each sync run to only records changed since the previous run.

WatermarkPolicy

schema.WatermarkPolicy = new WatermarkPolicy
{
    WatermarkMode        = "Timestamp",   // "Timestamp" | "Sequence" | "CompositeKey"
    WatermarkField       = "UpdatedAt",   // column that advances monotonically
    LastWatermarkValue   = null,          // null = start from beginning; engine updates this
    OverlapWindowSeconds = 300,           // re-scan last 5 min to catch late arrivals
    DedupeStrategy       = "LastWrite",   // "LastWrite" | "FirstWrite" | "None"
    ReplayEnabled        = true,          // idempotent: safe to re-run the same window

    // Optional: rule keys override the built-in filter/late-arrival logic
    FilterRuleKey        = "sync.watermark.filter",
    LateArrivalRuleKey   = "sync.watermark.late-arrival",
};

WatermarkMode values

ModeTypical field typeUse case
TimestampDateTime / DateTimeOffsetAudit timestamp, UpdatedAt, ModifiedAt
Sequencelong / intAuto-increment IDs, sequence numbers, offsets
CompositeKeyMultiple columnsChange-version tables with composite change keys

CdcFilterContext (output)

Returned by BuildCdcFilterContext — the runtime filter envelope passed to DataImportManager.

FieldTypeDescription
ResolvedFiltersList<AppFilter>Computed filter predicates using the watermark window
WindowStartobjectEffective lower-bound (includes overlap window)
WindowEndobjectEffective upper-bound (current watermark)
NewWatermarkValueobjectValue to be persisted as LastWatermarkValue after a successful run
LateArrivalActionstring"Include" / "Warn" / "Quarantine" per rule result

Conflict Resolution

Applies only when SyncDirection = "Bidirectional". When a record exists in both source and destination with different values, TryEvaluateConflictGate decides which value wins.

ConflictPolicy

schema.ConflictPolicy = new ConflictPolicy
{
    // Built-in rule keys (no rule engine required):
    //   "sync.conflict.source-wins"
    //   "sync.conflict.destination-wins"
    //   "sync.conflict.latest-timestamp-wins"
    //   "sync.conflict.fail-on-conflict"
    ResolutionRuleKey     = "sync.conflict.latest-timestamp-wins",

    CaptureEvidence       = true,    // populate BeepSyncManager.LastRunConflicts

    // Quarantine records whose conflict could not be resolved:
    QuarantineDsName      = "QuarantineDB",
    QuarantineEntity      = "ConflictQuarantine",

    MaxConflictsPerRun    = 500,
    OnMaxExceededAction   = "QuarantineRest",  // "Abort" | "Continue" | "QuarantineRest"
};

Built-in strategies

Rule keyBehaviour
sync.conflict.source-winsSource record always overwrites destination
sync.conflict.destination-winsDestination record is kept unchanged (default fallback)
sync.conflict.latest-timestamp-winsRecord with the later timestamp wins; requires WatermarkField
sync.conflict.fail-on-conflictAny conflict halts the run immediately
(custom key)Rule engine evaluates your custom rule; output must be "source", "destination", or "quarantine"

ConflictEvidence (audit)

// After a bidirectional run:
var conflicts = syncManager.LastRunConflicts;
Console.WriteLine($"Conflicts this run: {conflicts.Count}");

foreach (var ev in conflicts)
{
    Console.WriteLine($"  Key={ev.RecordKey}  " +
                      $"Resolution={ev.Resolution}  " +
                      $"Rule={ev.AppliedRuleKey}");
}

Retry & Checkpoint

Configure RetryPolicy to automatically re-attempt on transient errors and resume from a mid-run checkpoint without re-processing already-committed batches.

RetryPolicy

schema.RetryPolicy = new RetryPolicy
{
    MaxAttempts              = 4,
    BaseDelayMs              = 2000,
    BackoffMode              = "Exponential",  // "Fixed" | "Linear" | "Exponential"
    MaxResumeWindowHours     = 48,             // checkpoint invalidated after 48 h

    CheckpointEnabled        = true,           // save offset after each committed batch

    // Rule Engine key that classifies errors into categories
    ErrorCategoryRuleKey     = "sync.error.classify",

    NonRetryableCategories   = new List<string> { "Fatal", "Validation" },
};

Error categories

CategoryDefault actionDescription
TransientRetryNetwork timeout, transient DB lock
ValidationQuarantineSchema mismatch, constraint violation
ConflictRetryOptimistic-concurrency write conflict
FatalAbortAuthentication failure, data source not found

If the rule key ErrorCategoryRuleKey is not set or the rule engine is disabled, TryClassifyError applies keyword heuristics (e.g. "timeout" → Transient, "constraint" → Validation).

Checkpoint resume

// After a partial failure the checkpoint is stored in schema.ActiveCheckpoint.
// On the next call to SyncDataAsync, the manager calls IsCheckpointResumeSafe()
// which evaluates the rule "sync.checkpoint.resume-safe" or checks the age window.

// Force a full re-run by clearing the checkpoint:
schema.ActiveCheckpoint = null;
await syncManager.SyncDataAsync(schema, CancellationToken.None, progress);

Schema Governance & Promotion

PromoteSchemaAsync advances a schema through an approval lifecycle (e.g. Draft → Review → Approved). A rule engine gate can block promotion if the mapping quality or approval evidence is insufficient.

PromoteSchemaAsync

Task<SyncPreflightReport> PromoteSchemaAsync(
    DataSyncSchema    schema,
    string            targetState,    // e.g. "Review", "Approved", "Archived"
    string            approver   = null,
    string            changeNotes = null,
    CancellationToken token      = default
);

Promotion pipeline

#StepGate
1Null / empty checks on schema and targetStateHard fail
2Rule sync.schema.promotion-gate evaluated with schema, targetState, mappingApprovalState, requiredApprovalState contextConfigurable block
3Version artifact built (SHA hash of schema content, stamped with approver + notes)—
4Schema persisted via SaveSchemasAsync—

Example

var report = await syncManager.PromoteSchemaAsync(
    schema,
    targetState  : "Approved",
    approver     : "data-steward@example.com",
    changeNotes  : "Reviewed mapping completeness — 100%"
);

if (report.IsApproved)
    Console.WriteLine("Schema promoted to Approved.");
else
    foreach (var issue in report.Issues)
        Console.WriteLine($"  BLOCKED: {issue.Code} — {issue.Message}");

Examples

Eight runnable examples live in Editor/BeepSync/Examples/. Below is a representative selection.

Example 01 — Basic full sync

// Example01_BasicFullSync.cs
var syncManager = new BeepSyncManager(editor);
var schema = new DataSyncSchema
{
    Id                        = Guid.NewGuid().ToString(),
    SourceDataSourceName      = "SourceDB",
    DestinationDataSourceName = "DestDB",
    SourceEntityName          = "Customers",
    DestinationEntityName     = "Customers",
    SourceKeyField            = "CustomerId",
    DestinationKeyField       = "CustomerId",
    SyncType                  = "Full",
    SyncDirection             = "OneWay",
    BatchSize                 = 500,
};

var fieldHelper = new FieldMappingHelper(editor);
foreach (var m in fieldHelper.AutoMapFields("SourceDB","Customers","DestDB","Customers"))
    schema.MappedFields.Add(m);

syncManager.AddSyncSchema(schema);
await syncManager.SyncDataAsync(schema, CancellationToken.None,
    new Progress<PassedArgs>(p => Console.WriteLine(p.Messege)));

Console.WriteLine($"Status: {schema.SyncStatus}  LastSync: {schema.LastSyncDate:u}");

Example 03 — Incremental sync with watermark

// Example03_IncrementalSync.cs
var schema = new DataSyncSchema
{
    // ... source/dest/entity/key fields ...
    SyncType  = "Incremental",
    WatermarkPolicy = new WatermarkPolicy
    {
        WatermarkMode        = "Timestamp",
        WatermarkField       = "UpdatedAt",
        OverlapWindowSeconds = 300,
        DedupeStrategy       = "LastWrite",
        ReplayEnabled        = true,
    },
};

// First run — syncs records changed since epoch:
await syncManager.SyncDataAsync(schema, CancellationToken.None, null);

// Second run — only picks up records changed since the first run:
await syncManager.SyncDataAsync(schema, CancellationToken.None, null);

Example 04 — Bidirectional conflict resolution

// Example04_BidirectionalConflict.cs
schema.SyncDirection  = "Bidirectional";
schema.WatermarkPolicy = new WatermarkPolicy
    { WatermarkMode = "Timestamp", WatermarkField = "ModifiedAt", ReplayEnabled = true };

schema.ConflictPolicy = new ConflictPolicy
{
    ResolutionRuleKey   = "sync.conflict.latest-timestamp-wins",
    CaptureEvidence     = true,
    QuarantineDsName    = "QuarantineDB",
    QuarantineEntity    = "ConflictQuarantine",
    MaxConflictsPerRun  = 500,
    OnMaxExceededAction = "QuarantineRest",
};

await syncManager.SyncDataAsync(schema, CancellationToken.None, null);

foreach (var ev in syncManager.LastRunConflicts)
    Console.WriteLine($"Key={ev.RecordKey}  Resolution={ev.Resolution}");

Example 05 — Retry with checkpoint resume

// Example05_RetryAndCheckpoint.cs
schema.RetryPolicy = new RetryPolicy
{
    MaxAttempts           = 4,
    BaseDelayMs           = 2000,
    BackoffMode           = "Exponential",
    MaxResumeWindowHours  = 48,
    CheckpointEnabled     = true,
    ErrorCategoryRuleKey  = "sync.error.classify",
    NonRetryableCategories = new List<string> { "Fatal", "Validation" },
};

// Run 1 — saves a checkpoint if it fails mid-way
await syncManager.SyncDataAsync(schema, CancellationToken.None, progress);

// Run 2 — resumes from the checkpoint automatically if it is still within 48 h
await syncManager.SyncDataAsync(schema, CancellationToken.None, progress);

// Force full re-run:
schema.ActiveCheckpoint = null;
await syncManager.SyncDataAsync(schema, CancellationToken.None, progress);
